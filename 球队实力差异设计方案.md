# 球队实力差异设计方案（速度差异）

> 目标：让强队打弱队时，强队获胜概率高于 50%，同时保留爆冷可能，并且观感物理直觉自然。
> 本文档只描述方案，暂不含代码实现。

## 1. 背景与问题

当前 `js/engine.js` 的比赛引擎里，两颗球在圆形场地内以**恒定速率**乱弹，谁的球穿过旋转的球门弧就得分。整个过程对双方完全对称，因此无论强弱，两队进球期望都是 50/50。用户希望引入球队实力差异，让强弱对战更接近真实赛事。

曾考虑过的两种方案：

- **进球闸门方案**（球过线后再用概率判定是否算进球）：球都过门线了还要“骰子”决定，视觉上很假，**否决**。
- **转向偏置方案**（每帧把球速方向朝球门中心掰一点）：效果强，但球会“自己找门”、中途变向，物理上不直觉，观感偏假，**否决**。

**已确定采用速度差异方案。**

## 2. 核心思路

不改变“球过线=进球”这条铁律，也不引入任何方向偏置，只让**强队球的目标速率更高**：

- 强队球更快 -> 单位时间内飞过更长路程 -> 更频繁接触外圈（进球判定区）-> 进球机会更多。
- 弱队球更慢 -> 撞墙次数少 -> 进球机会少，但仍会自然地偶尔进球。
- 两队都遵守“恒定速率 + 真实穿过门线”的物理，没有“突然变向”，观感直觉自然。

> 物理依据：单球测试证实进球数与速率近似线性正相关（速率 384 -> 每场 0.62 球；速率 576 -> 每场 0.97 球）。速率差异能稳定转化为进球期望差异。

### 相关引擎位置（现状）

- `integrate(h)`（`js/engine.js` 约 285 行）：`b.x += b.vx*h` 后把速度重新归一化到全局 `TARGET_SP=480`。**速度差异就改这里**--每颗球归一化到自己的 `b.sp`，而非全局 `TARGET_SP`。
- 计分：`wallAndGoal()` 中球进门口且 `dist >= R` 时调 `scoreGoal(b)`（约 330 行）。这条逻辑**完全不动**。
- 两颗球：`state.balls[0]` 为主队（tA，红），`state.balls[1]` 为客队（tB，蓝），`b.ref` 指向球队对象。
- 球门在旋转：`state.goalCenter + state.rot`。

## 3. 速度差异的实现要点

每颗球新增字段 `b.sp`（该球的目标速率）。每帧 `integrate` 归一化时用 `b.sp` 替代全局 `TARGET_SP`：

1. `newMatch` 时由 `computeStrength(tA, tB)` 算出两队速率 `[spA, spB]`，分别赋给 `balls[0].sp` / `balls[1].sp`。
2. `integrate` 归一化：`b.vx = b.vx/sp * b.sp`（`sp = Math.hypot(b.vx,b.vy)`），保持各球速率恒定在各自 `b.sp`。
3. `resetBallsToCenter`（开球归位）也用 `balls[i].sp` 而非全局 `TARGET_SP` 重置速度。
4. `makeBall` 初速度用 `b.sp`。

关键点：**只改每球的目标速率、不改方向、不改进球判定**。势均力敌时两队 `sp` 都等于 `BASE`（=原 `TARGET_SP`），与原引擎完全一致；有差距时强队快、弱队慢。

> 注意：两球速率不同，并排看会有“一快一慢”的视觉差异。这是方案固有特征（用速率差而非方向偏置表达强弱），相比“球中途变向”更物理直觉，但视觉公平感不如完全对称。可接受程度见第 5 节权衡。

## 4. 实力评分（rating）来源

引擎通过 `TEAM.rating(t)` 读取球队实力；**若该函数不存在或两队 rating 相等，则退回 50/50**（两队 `sp` 都 = `BASE`，与原引擎一致）。世界杯页即如此。各页面适配器分别提供：

- **联赛页**（`js/badge.js` 的 `makeBadgeTeamAdapter` + `js/standings.js`）：用最近赛季积分 `pts`（掺净胜球 `gf-ga` ×0.1 微调）。数据在页面内联的全局 `SEASONS[0].clubs`（结构 `{code,pld,w,d,l,gf,ga,pts}`），`CLUBS[code]` 是队对象。让 `makeBadgeTeamAdapter` 接收可选 `rating` 回调；`standings.js` 建 `ratingMap`，赛季切换时重建，空赛季（全 0）回退到最近有数据赛季。
- **欧冠页**（`js/ucl.js`）：用联赛阶段排名数组 `LEAGUE`（`{code,pts,...}`）里的积分 + 净胜球微调。
- **世界杯页**（`js/bracket.js`，国旗版 TEAM）：无实力数据，不提供 `rating` -> 自动退回 50/50。

## 5. 概率与手感设计

把两队 rating 差映射成强队的“进球份额” `sAp`，再映射成速率差：

- **Elo logistic 映射**：`sA = 1 / (1 + 10^(-(Ra - Rb)/D))`。`D` 是主手感旋钮，越大越接近 50/50，越小越拉开差距。
- **冷门旋钮 `UPSET`（0~1）**：`sAp = lerp(sA, 0.5, UPSET)`，把份额往 0.5 拉，保留爆冷空间。
- **不对称度**：`asym = |sAp - 0.5| * 2`（0=势均力敌，1=最悬殊）。势均力敌时 `asym=0` -> 两队速率都 = `BASE`，视觉与原引擎完全一致。
- **速率映射**：强队 `sp = BASE + SPREAD * asym`，弱队 `sp = BASE - SPREAD * asym`。`SPREAD` 控制最大速率差。
- 两队速率关于 `BASE` 对称，平均速率恒为 `BASE`，不改变整体比赛节奏。

### 默认取值（保守起步）

| 常量 | 默认 | 说明 |
|---|---|---|
| `RATING_D` | `40` | Elo 分母。越大越接近 50/50，越小差距拉得越开。 |
| `UPSET` | `0.30` | 冷门旋钮（0~1）。把强队份额往 0.5 拉，保留爆冷。 |
| `BASE` | `480` | 基础速率，等于原 `TARGET_SP`。势均力敌时两队都用此速率。 |
| `SPREAD` | `96` | 最大速率差幅度。强队最快 `BASE+SPREAD`、弱队最慢 `BASE-SPREAD`。 |
| `PEN_RATING_AMP` | `0.15` | 点球命中率按实力份额浮动幅度（次要）。 |

常量集中放在配置区（`TARGET_SP` 附近）便于调参。

### 实测验证与权衡（重要）

本方案在当前引擎（每场进球率 < 1 球）下**效果天然偏弱**：速率差异是“量”的调节（多撞几次墙，每次仅约 5% 进门概率），产生的期望进球差很小，易被随机性淹没。以下为 Node 模拟（复刻物理 + 进球后开球重置，各档 400~600 局）的近似胜率（A胜/B胜/平）：

| SPREAD | 视觉速率差 | 均势 60v60 | 中差 80v60 | 大差 94v40 | 极大 94v20 |
|---|---|---|---|---|---|
| 48 (±10%) | 轻微 | ~50/50 | ~50/50 | ~50/50 | ~50/50 |
| 96 (±20%) | 可辨 | 34/33/33 | 37/32/31 | 38/30/32 | 42/26/32 |
| 144 (±30%) | 明显 | 36/34/31 | 37/32/31 | 44/26/30 | 48/23/29 |
| 192 (±40%) | 悬殊 | 36/30/34 | 40/29/31 | 45/21/34 | 46/19/35 |

**结论与权衡**：

- 即便极限 `SPREAD=192`（±40%，弱队球明显偏慢、视觉不公平），极大差对阵也仅约 46% 胜，**无法达到“强队明显 favored”**。
- `SPREAD=96`（±20%，速率 576/384）为保守起步：视觉差异可辨但不突兀，强队胜率提升有限（极大差约 42%）。若觉效果太弱可逐步加大 `SPREAD`，代价是两球快慢差异更明显。
- 相比转向偏置方案（同对阵大差可达 84% 胜），速度方案效果弱很多，但胜在物理直觉、无变向。**这是方案固有的效果-观感权衡，已知并接受。**
- 如后续发现效果过弱无法满足需求，可参考第 7 节备选方案（转向偏置）或考虑混合方案。

## 6. 点球大战（次要）

点球命中率 `PEN_GOAL_P = 0.72` 目前两队相等。可按实力份额微调（强队命中率略高、弱队略低），以 `PEN_GOAL_P` 为中心、`PEN_RATING_AMP` 为幅度浮动，夹在 [0.5, 0.9]。优先级低于常规时间的速率差异。

## 7. 备选方案（未采用，仅备忘）

- **转向偏置**（每帧把球速方向朝球门中心掰一点，只改方向不改速率）：效果强（大差 84% 胜），且两球速率一致、视觉对称；但球会“自己找门”、中途变向，物理直觉偏假。若速度方案效果过弱无法接受，可回退到此方案。
- **进球闸门**（球过线后概率判定）：最假，已否决。

## 8. 落地步骤

1. 在 `engine.js` 配置区（`TARGET_SP` 附近）新增 `RATING_D` / `UPSET` / `BASE` / `SPREAD` / `PEN_RATING_AMP` 常量；`TARGET_SP` 保留为 `BASE` 的别名或直接被 `BASE` 取代。
2. 新增 `computeStrength(tA, tB)`：读 `TEAM.rating`，算 `sAp` 与 `[spA, spB]`；无 rating / 缺失 / 相等 -> `[BASE, BASE]`。
3. `makeBall` 加 `sp` 字段；`newMatch` 调 `computeStrength` 赋值；`integrate` 与 `resetBallsToCenter` 归一化到 `b.sp`。
4. 给三处 TEAM 适配器补 `rating(t)`：`badge.js`/`standings.js`、`ucl.js`、`bracket.js`（世界杯不补，自动退回）。
5. 可选：`PEN_GOAL_P` 按实力份额微调。
6. 浏览器验证三个页面（`leagues/league.html?c=spain`、`cups/ucl.html`、`cups/worldcup.html`）：跑多局确认强队胜率略高于 50% 且无报错；按第 5 节数据按需调 `SPREAD`。
7. 更新 README（配置项表格补 `BASE` / `SPREAD` / `RATING_D` / `UPSET` 与实力机制说明，并注明速度方案的效果-观感权衡）。
